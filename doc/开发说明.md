## For Agent

### 运行方法

首次安装:

```powershell
cd D:\checker
python -m venv venv
.\venv\Scripts\Activate.ps1
python -m pip install -r sidecar\requirements.txt

cd app
npm install
```

首次运行前复制配置:

```powershell
cd D:\checker
Copy-Item config.example.toml config.toml
```

编辑 `config.toml`:本地扫描可以先留空;腾讯文档读写填 `[tencent_docs]` + `[user].reviewer_name`;Agent 填 `[llm]`。

启动:

```powershell
cd D:\checker\app
npm run dev
```

常用检查:

```powershell
cd D:\checker
.\venv\Scripts\Activate.ps1
pytest sidecar/tests/

cd app
npm run typecheck
```

需要单独调 sidecar API / 看 OpenAPI 时,另开一个终端:

```powershell
# 单独跑 sidecar(独立调 API / 看 OpenAPI)
cd D:\checker
.\venv\Scripts\Activate.ps1
python -m sidecar.serve          # http://127.0.0.1:8775/docs
```

启动排错:

- macOS / Linux 的 venv 激活命令是 `source venv/bin/activate`。
- 如果 npm 报 `EPERM ... npm-cache`,改用仓库内缓存再执行命令:`npm --cache D:\checker\tmp\npm-cache run dev`。
- 如果启动时报 `better-sqlite3 ... NODE_MODULE_VERSION ...`,说明原生模块按普通 Node 编译了,在 `app/` 下执行 `npm exec electron-builder -- install-app-deps` 后重启。`npm install` 后会通过 `postinstall` 自动跑这一步。
- 开发模式 Electron 的用户数据目录固定到 `tmp/electron-user-data`,避免 Windows `%APPDATA%` 权限/缓存目录问题影响 DB 初始化。
- 配置读取顺序:`CHECKER_CONFIG` → 仓库根 `config.toml` → 平台配置目录。

### 协作偏好
- 若当前解决了一个 TODO，记得在本开发说明里打勾
- **和秘钥、用户隐私相关内容不要进 git 跟踪**（重要！）
- **不要主动加注释 / 文档 / 兼容层 / 错误处理**,除非任务要求。三行相似代码好过一个过早抽象。
- 工具自由度 = 人类:`RenameOp` / `DeleteOp` / `MoveOp` agent 直接构造,不强制 propose-only,安全靠 confirm 卡 + 服务端哈希校验兜底。
- SOP / workflow 文档是**写给 agent 看的**,不是讨论笔记 —— 不要写 GA 参考 / 待补 / 无效边界,一次推一态。
- 不能踩的雷集中在 `CLAUDE.md` "critical invariants" 节(身份隐藏 / `_rows` 不给 agent / confirm 流程 / 列号写死 / 规则集中在 logic_checker),改 sidecar 前过一遍。

### Git 提交流程

**默认走 PR,不要直接把改动推到主分支。** 提交前先看远端有没有更新;如果远端领先,先并到本地并把冲突处理完,不要把冲突留给仓库管理者。

约束:

- 在 feature 分支上提交改动,推到自己的远端分支后提 PR。
- 提交前先同步远端;若有更新,先 merge 到本地并解决冲突,验证通过后再 push。
- 同步远端时留意之前的 PR 是否已经被合并;如果已合并,先基于最新 `audit-copilot` 整理本地分支,不要继续在过期分支上叠提交。
- PR 提到 `audit-copilot`。
- **不要 force push**(尤其 `main` / `audit-copilot`),不要 `--no-verify` 跳 hook,不要 `git reset --hard` 别人的提交。
- **只 commit 用户明确要求 commit 的改动**,不要主动 commit。
- 遇到 merge 冲突默认解决而非丢弃,冲突文件可能是用户的在途工作，复杂情况记得问用户。
