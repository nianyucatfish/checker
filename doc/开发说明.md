## For Agent

### 运行方法

前置一次性:

```bash
# 1. sidecar venv
python -m venv .venv
.venv\Scripts\activate          # Windows;mac/linux 用 source .venv/bin/activate
pip install -r sidecar/requirements.txt

# 2. 前端依赖
cd app && npm install

# 3. 配置
cp config.example.toml config.toml   # Windows: copy
# 编辑 config.toml 填上面表里需要的字段
```

日常开发:

```bash
# 前端 + Electron(Electron main 会自动 spawn sidecar 子进程,无需另起)
cd app && npm run dev

# 单独跑 sidecar(独立调 API / 看 OpenAPI)
python -m sidecar.serve          # http://127.0.0.1:8775/docs

# 跑测试(目标全绿,目前 71 passed)
pytest sidecar/tests/

# 前端类型检查
cd app && npm run typecheck

# 打包
cd app && npm run build
```

### 协作偏好
- 若当前解决了一个 TODO，记得在本开发说明里打勾
- **和秘钥、用户隐私相关内容不要进 git 跟踪**（重要！）
- **不要主动加注释 / 文档 / 兼容层 / 错误处理**,除非任务要求。三行相似代码好过一个过早抽象。
- 工具自由度 = 人类:`RenameOp` / `DeleteOp` / `MoveOp` agent 直接构造,不强制 propose-only,安全靠 confirm 卡 + 服务端哈希校验兜底。
- SOP / workflow 文档是**写给 agent 看的**,不是讨论笔记 —— 不要写 GA 参考 / 待补 / 无效边界,一次推一态。
- 不能踩的雷集中在 `CLAUDE.md` "critical invariants" 节(身份隐藏 / `_rows` 不给 agent / confirm 流程 / 列号写死 / 规则集中在 logic_checker),改 sidecar 前过一遍。

### Git 提交流程

**每次向仓库提交时,先同步上游再提交本地改动。** 不要直接在落后于 remote 的分支上 push,会触发 force 才能推上去的尴尬场面。

标准流程(在 feature 分支或 `v2` 上):

```bash
# 1. 看清楚本地状态
git status
git diff

# 2. 拿到 remote 最新
git fetch origin

# 3. 把上游 merge 进本地分支(不要 rebase 已 push 出去的提交)
git merge origin/<current-branch>
# 如果有冲突:解决冲突 → git add <files> → git commit(走默认 merge commit message 即可)

# 4. 本地改动正常 add + commit
git add <具体文件,不要 git add -A>
git commit -m "<type>(<scope>): <简述>"

# 5. push
git push origin <current-branch>
```

约束:

- **PR 提到 `main`**,主开发在 `v2`。
- **不要 force push**(尤其 `main` / `v2`),不要 `--no-verify` 跳 hook,不要 `git reset --hard` 别人的提交。
- **只 commit 用户明确要求 commit 的改动**,不要主动 commit。
- 遇到 merge 冲突默认解决而非丢弃,冲突文件可能是用户的在途工作，复杂情况记得问用户。
